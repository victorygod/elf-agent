# Elf 工作流设计文档

> **状态：暂缓**
>
> 经过讨论，我们明确了工作流编辑器的核心目标实际上是**方便创建 Agent**，而非构建通用的 DAG 编排系统。当前 Elf Agent 的结构（单 LLM 循环 + 工具调用）不需要可视化画布来编排——拓扑简单到在脑子里就能想清楚。真正降低 Agent 创建门槛的方式是 shared/ 模块化封装 + 一个专门创建 Agent 的 Agent（elf-003），通过对话生成配置和代码，而非拖拽控件。
>
> 本文档中的设计结论（万物皆节点的 Keras 哲学、运行时探查、对话式创建）仍然有参考价值，待未来 Agent 结构复杂到需要可视化编排时再启动。当前优先级转向 [elf-003 设计文档](./elf-003-design.md)。

## 一、项目愿景

构建一个基于 Web 的工作流编排工具，让用户既能通过自然语言与 AI 协作搭建工作流，又能像操作成熟软件一样在界面上直接拖拽、点选、修改，使工作流成为一种介于"纯代码"和"成品应用"之间的灵活中间状态，适配不同技术背景的人群。

## 二、核心交互机制

系统提供"对话式编程"与"可视化编辑"两种平等且互通的操作路径。用户可在对话面板中用日常语言描述需求，Agent 据此生成完整的工作流定义（节点、连线、初始代码）；同时，用户也能在画布上自由增删节点、拖拽连线、双击节点打开内置的代码编辑器（Monaco）直接修改任意逻辑。两种操作实时同步，互不覆盖。

## 三、节点设计哲学：万物皆节点

### 3.1 设计理念

采纳 Keras Layer 的对象哲学：**万物皆节点，节点皆可调用**。

Keras 的设计要点：

- **万物皆层**：没有"计算层"和"配置层"的二元划分。`Dense(64)` 是层，`Dropout(0.5)` 是层，`Flatten()` 也是层。哪怕 Dropout 在推理时什么都不做，它仍然是一个对等的层对象。
- **构造配置行为，调用执行计算**：`dense = Dense(64, activation='relu')`（构造：指定是什么）→ `output = dense(input)`（调用：数据流过）。构造参数定义"这个节点是什么"，调用执行定义"这个节点对数据做什么"。
- **可调用是唯一接口**：不管内部多复杂——`LSTM` 里有门控循环，`KSampler` 里有 50 步采样——对外都是一个 `call(x)`。调用者不需要知道内部结构。
- **组合是唯一的编排方式**：`x = Dense(64)(input); x = Dropout(0.5)(x); output = Dense(10)(x)`。没有两套交互——只有一种操作：把 A 的输出接到 B 的输入。

### 3.2 对应设计约束

- **不存在"计算节点"和"配置节点"的二元划分**：术语表是一个节点，模型配置是一个节点，翻译也是一个节点——它们都是同一种东西
- **构造参数可以在面板编辑，也可以被连线值覆盖**：构造参数本质是输入端口的默认值
- **引擎只调度，不理解节点语义**：引擎知道每个节点有 inputs、outputs、一个 `run` 方法，不关心也不应该关心节点内部在做什么
- **只有一种基类**：不需要 `ComputeNode` / `ConfigNode` / `DataNode` 的继承体系

## 四、节点定义

### 4.1 节点模型

一个节点 = 一个 Keras 式的可实例化对象：

```
┌────────────────────────────────────────┐
│               TranslateNode             │
│                                         │
│  输入端口（来自其他节点的连线）            │
│  ┌──────────────────────────────────┐   │
│  │ segments: Segment[]               │   │
│  │ glossary: Record<string,string>   │   │
│  │ modelConf: { model, url, token }  │   │
│  │ contextPolicy: { windowSize }     │   │
│  └──────────────────────────────────┘   │
│                                         │
│  输出端口                                │
│  ┌──────────────────────────────────┐   │
│  │ translated: string[]              │   │
│  └──────────────────────────────────┘   │
│                                         │
│  运行时状态（可探查）                     │
│  ┌──────────────────────────────────┐   │
│  │ 进度: 第 6/203 段                  │   │
│  │ 当前段译文: "阿特雷迪斯公爵走进..."    │   │
│  │ 术语表条目数: 23                   │   │
│  │ 上下文历史: [user, assistant, ...] │   │
│  └──────────────────────────────────┘   │
│                                         │
│  内部代码（双击打开 Monaco）              │
│  ┌──────────────────────────────────┐   │
│  │ async function run(input, ctx) {  │   │
│  │   // 任意 JavaScript 代码          │   │
│  │   // 可 require/import 任意库      │   │
│  │ }                                 │   │
│  └──────────────────────────────────┘   │
└────────────────────────────────────────┘
```

### 4.2 节点生命周期

```
创建      节点对象 = new TranslateNode({ model: 'deepseek', contextWindow: 6 })
连线      画布上 segments → translate → translated
运行      引擎调 translate.run({ segments: [...], ... }, context)
探查      运行时点开节点看当前段译文 + glossary 状态 + 上下文历史
替换      拆掉连线，换成 GLMTranslateNode，接口相同
```

### 4.3 代码约束

所有节点的 `run` 方法遵循统一签名：

```js
async function run(input, context) {}
```

- **input**：引擎自动合并上游各端口输出后传入的对象。每个 key 对应一个输入端口，值来自连线，无连线时回退到构造参数默认值
- **context**：引擎提供的运行时工具箱，挂载了全局状态存储、日志打印、进度上报（`reportProgress`）、取消信号（`AbortSignal`）以及重试辅助工具

节点执行后必须返回一个与预定义输出端口匹配的普通对象。

**除此之外，节点内部可使用任意 JavaScript 语法，允许自由 require 或 import 外部依赖库，不设安全限制以换取最大实现自由度。**

## 五、运行时展示

### 5.1 画布上的状态可视化

- 每个节点标注执行状态（等待中 / 运行中 / 成功 / 失败），用颜色区分
- 运行中的节点显示进度（如"第 6/203 段"）
- 出错节点红色高亮，点击展开错误日志和出错时的输入数据现场

### 5.2 节点状态探查

点击运行中的节点，展示当前内部状态快照：

- 翻译节点：当前段的译文预览、术语表当前内容、上下文历史最近几轮
- KSampler 节点：中间 latent 预览
- 任何节点：已消费的 input 数据、已产出的 output 数据

### 5.3 错误日志

节点运行失败时，点开可直接查看错误堆栈、出错时的 input 数据现场。不需要离开画布去查日志文件。

## 六、循环与流程控制

为避免破坏 DAG 结构，循环逻辑优先在单个节点的代码内部通过 `for` / `while` 实现，执行过程中节点可通过 `reportProgress` 向前端推送进度，并通过检查 `AbortSignal` 响应中断请求，防止死循环。

仅当用户需要在画布上显式展示循环内多步骤协作时，才引入"迭代器节点"（内部封装子工作流），但绝对禁止在连线层面形成物理闭环。

## 八、当前 Agent 系统与工作流的关系

当前 Elf 的单 Agent 聊天系统可视为工作流的一个特例：一个 Agent 节点 + 模型配置节点 + 聊天面板（实时 input/output）。

现有 Agent 的配置（`config.json`）在迁移到工作流后，对应一个节点实例的构造参数。现有 Agent 的聊天面板在工作流视图中缩为侧栏。
